AI-Tester 平台简介

AI-Tester 是一个与 AI Coding Agent 配合使用的自动化 API 测试平台，帮助您快速构建、管理和执行 API 测试。

核心能力

  • 项目管理 — 以项目为维度组织测试资源，支持多环境配置
  • 测试用例编写 — 可视化步骤编辑器，支持 HTTP 请求、断言、变量提取、子用例调用、数据集加载 5 种步骤类型
  • 测试套件 — 将多个用例组织为套件，一键批量执行
  • 执行与结果 — 实时查看执行状态、请求/响应详情、断言结果
  • 数据集 — 管理测试数据，驱动数据驱动的测试场景
  • AI 自动生成 — 配置 LLM 后，基于 API 知识自动生成测试用例

用例基本信息

  • 名称（必填）：用例的标识名称
  • 模块：路径字符串（如 user/auth），用于树形组织，不是独立实体
  • 优先级：critical high medium low
  • 标签：逗号分隔的字符串，用于横向分类过滤

五种步骤类型

1. HTTP 请求步骤

发送一次 HTTP 请求，是最常用的步骤类型。

  • 请求方法：GET / POST / PUT / PATCH / DELETE / HEAD / OPTIONS
  • URL：支持模板变量，如 {{baseUrl}}/api/users/{{userId}}
  • 请求体：JSON 格式，支持模板变量
URL 和 Body 中的 {{variableName}} 会在执行时从上下文变量中解析替换。

2. 断言步骤

对上一步 HTTP 响应或上下文变量执行断言验证。

  • 断言来源：状态码 / 响应头 / 响应体 / JSONPath / 上下文变量
  • 操作符：等于、不等于、包含、不包含、大于、大于等于、小于、小于等于、正则匹配、存在、不存在、类型为
  • 选择 JSONPath / 响应头 / 上下文变量 时，会显示「表达式」输入框

3. 变量提取步骤

从 HTTP 响应中提取值存入上下文变量，供后续步骤使用。

  • 提取来源：完整响应体 / JSONPath / 响应头 / 状态码 / 正则表达式
  • 提取的变量可在后续步骤中通过 {{variableName}} 引用

4. 子用例调用步骤

调用另一个测试用例，被调用用例与调用方共享执行上下文。

被调用用例与调用方共享同一个 RunContext，变量双向可见。

5. 加载数据集步骤

将数据集加载到上下文变量中，用于数据驱动测试。

加载后通过 {{variableName}} 访问行数组，通过 {{variableName[0].field}} 访问具体字段。

通用步骤选项

每个步骤都有以下通用选项：

  • 失败后继续：开关，控制该步骤失败后是否继续执行后续步骤
  • 重试次数：0~3 次，步骤失败后自动重试

步骤视图切换

用例编辑器提供两种视图模式：

  • 表单视图：逐步骤表单编辑，适合精细配置
  • 流水线视图：可视化流程图，适合查看整体流程

用例操作

  • 克隆：复制一个完全相同的用例
  • 搜索：支持按名称搜索用例

执行与结果

测试执行是验证 API 正确性的核心环节。AI-Tester 支持一键触发套件执行，并详细记录每个步骤的执行过程。

业务流程

业务流程
graph TD
    A[在套件页面触发运行] --> B[选择执行环境]
    B --> C[系统创建 TestRun]
    C --> D[依次执行套件中的用例]
    D --> E[每个用例依次执行步骤]
    E --> F[记录每步的请求/响应/断言/提取]
    F --> G[汇总结果]
    G --> H[查看运行列表]
    H --> I[点击进入运行详情]
    I --> J[查看用例结果]
    J --> K[展开查看步骤详情]

运行列表

运行列表页面展示所有历史执行记录，包含以下信息：

运行详情

点击某条运行记录可进入详情页，查看完整的执行信息：

  • 摘要卡片：总用例数、通过数、失败数、通过率
  • 元信息：环境、触发方式、耗时、开始时间
  • 用例结果列表：每个用例的通过/失败状态

步骤结果详情

展开用例结果后，可查看每个步骤的详细信息，通过标签页切换：

  • 概览：步骤类型、状态、耗时、提取的变量
  • 请求（HTTP 步骤）：完整请求信息（method / url / headers / body）
  • 响应（HTTP 步骤）：响应状态码、耗时、响应体
  • 断言（断言步骤）：操作符、期望值、实际值、是否通过
  • 错误（失败步骤）：错误信息和堆栈
失败用例默认展开步骤详情，方便快速定位问题。

AI 设置

配置 LLM 服务提供商和 API Key，这是使用 AI 功能（知识解析、用例生成）的前提。

业务流程

业务流程
graph LR
    A[进入 AI 设置页] --> B[选择 Provider]
    B --> C[输入 Model 名称]
    C --> D[输入 API Key]
    D --> E[配置参数]
    E --> F[保存配置]
    F --> G[测试连接]
    G --> H{连接成功?}
    H -->|是| I[✅ 可使用 AI 功能]
    H -->|否| J[检查配置后重试]

操作步骤

1

选择 Provider

从下拉菜单选择 LLM 提供商：

  • OpenAI：支持 gpt-4o、gpt-4o-mini 等模型
  • Anthropic：支持 claude-sonnet-4 等模型（预留）
  • Custom：兼容 OpenAI API 格式的自定义端点（如 Ollama、vLLM）
2

输入 Model 和 API Key

Model 输入框支持预设建议（如 gpt-4o），也可手动输入。API Key 以密码形式输入，保存后加密存储。

3

配置参数（选填）

  • Base URL：仅 Custom 模式需要，填写兼容端点地址
  • Temperature：0~2，控制输出随机性（默认 0.7）
  • Max Tokens：100~128000，控制最大输出长度（默认 4096）
4

保存并测试

点击「保存」后，可点击「测试连接」按钮验证 LLM 服务是否可用。连接成功后即可使用 AI 知识解析和用例生成功能。

API Key 使用 AES-256-GCM 加密存储在数据库中，确保密钥安全。

4 种录入方式

1. 手动添加端点

适合少量端点的精确录入。点击「添加端点」按钮，在对话框中填写：

2. 导入 OpenAPI 文档

适合已有 OpenAPI/Swagger 文档的项目。点击「导入 OpenAPI」，粘贴 JSON/YAML 格式的文档内容，系统自动解析并批量创建端点。

支持 OpenAPI 3.x 和 Swagger 2.0 格式。

3. 解析 cURL 命令

适合从浏览器开发者工具复制的请求。点击「解析 cURL」，粘贴 cURL 命令，系统自动解析出请求方法、URL、Headers 等信息。

4. AI 解析文本

适合没有 API 文档的场景。用自然语言描述你的 API，AI 会自动解析为结构化端点。

1

输入描述文本

例如："我的用户服务有注册接口 POST /api/register，登录接口 POST /api/login，获取用户信息 GET /api/users/:id，需要 Bearer Token 认证"

2

AI 解析预览

系统调用 LLM 解析，返回端点列表预览，每个端点显示方法、路径和摘要。

3

勾选并添加

勾选需要的端点，点击「添加选中项」入库。

AI 解析文本功能需要先在 AI 设置中配置好 LLM 服务。

知识库表格

所有端点以表格形式展示，包含以下列：

  • 方法：彩色标签（GET POST PUT DELETE）
  • 路径：API 路径
  • 摘要：接口说明
  • 认证：认证方式
  • 来源：manual / openapi / curl / text

3 步向导

Step 1 — 选择端点

从知识库中选择要生成用例的 API 端点。支持全选/取消全选，每个端点显示方法标签、路径和摘要。

POST /api/auth/login 用户登录
GET /api/users/:id 获取用户信息
DELETE /api/users/:id 删除用户
已选 2 个端点

Step 2 — 选择策略

选择用例生成策略，不同策略关注的测试角度不同：

正向用例

验证 API 正常功能路径

异常用例

测试错误输入和边界情况

认证用例

测试认证和授权场景

综合用例

覆盖以上所有场景

还可选填自定义 Prompt，给 AI 额外的生成指令（如"重点关注分页参数的边界值"）。

Step 3 — 审核确认

AI 生成完成后，展示预览列表。每个用例卡片显示：

  • 用例名称、优先级、标签
  • AI 推理说明（reasoning）
  • 展开可查看具体步骤列表
登录接口 - 正常登录 high auth 3 steps ▾

验证使用正确的用户名和密码可以成功登录

勾选需要的用例后，点击「确认选中项」入库。入库后可在测试用例页面查看和编辑。

生成的用例是预览状态，不会直接入库。只有经过你审核确认后才会创建为正式用例。
AI 生成需要消耗 LLM API Token，请关注 Token 用量。生成耗时可能较长（30~120 秒），请耐心等待。
